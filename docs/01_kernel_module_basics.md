# 📚 [이론] 커널 모듈의 생명 주기와 커널 로그

커널(Kernel)은 운영체제의 심장이다. 일반적인 애플리케이션(Application)과 달리, 커널 코드는 함부로 멈추거나 재시작할 수 없다. 그렇다면 새로운 디바이스 드라이버나 기능을 커널에 추가하려면 시스템을 매번 재부팅해야 할까? 
이를 방지하기 위해 등장한 개념이 바로 **적재 가능한 커널 모듈 (LKM: Loadable Kernel Module)** 이다.

## 1. 커널 모듈의 구조 (생명 주기)
일반 C 프로그램은 `main()` 함수에서 시작하지만, 커널 모듈은 `main()`이 없다. 대신 시스템에 들어갈 때와 나갈 때 호출되는 두 가지 핵심 함수로 구성된다.

* **`module_init()` 매크로:**
  * 모듈이 커널에 적재될 때(`insmod`) 최초로 한 번 실행되는 초기화 함수를 지정한다.
  * 현업 예시: 하드웨어 장치의 메모리를 할당하거나, 디바이스 드라이버 번호를 커널에 등록할 때 사용.
* **`module_exit()` 매크로:**
  * 모듈이 커널에서 제거될 때(`rmmod`) 실행되는 정리 함수를 지정한다.
  * 현업 예시: 사용했던 메모리를 반환하고 등록했던 장치를 해제하여 메모리 누수(Memory Leak)를 방지.

## 2. 커널 공간에서의 출력: `printk` (또는 `pr_info`)
* 유저 공간(User Space)의 애플리케이션은 화면에 글자를 띄울 때 `printf`를 사용하지만, 커널 공간(Kernel Space)은 C 표준 라이브러리(`libc`)를 사용할 수 없다.
* 대신 커널 전용 출력 함수인 `printk()` (최신 커널에서는 로그 레벨이 지정된 `pr_info`, `pr_err` 등을 권장)를 사용한다.
* 이 메시지들은 터미널에 바로 뜨지 않고 커널 내부의 링 버퍼(Ring Buffer)라는 메모리에 저장되며, 우리는 **`dmesg`** 명령어를 통해 이 버퍼를 읽어 화면에 출력할 수 있다.

## 3. 현업 디버깅 마인드셋
디바이스 드라이버를 개발할 때 드라이버가 로드되지 않는다면, 가장 먼저 해야 할 행동은 화면을 쳐다보는 것이 아니라 **무조건 `dmesg` 쳐서 커널이 뱉어낸 에러 로그부터 확인하는 것**이다.
