# 📚 [이론] Cross-Compile 파이프라인의 핵심 구조와 동작 원리

임베디드 리눅스 개발 환경에서는 타겟 보드의 부족한 리소스(CPU, RAM) 한계를 극복하기 위해, 강력한 호스트 PC에서 소프트웨어를 빌드하는 크로스 컴파일 방식이 필수적이다.

## 1. `Kbuild` 시스템과 크로스 컴파일러의 연동 메커니즘
리눅스 커널 빌드 시스템(`Kbuild`)은 C 소스 코드 파일(`.c`)만으로 빌드를 수행할 수 없다. 빌드 과정에서 호스트 환경의 아키텍처를 타겟 환경으로 속이기 위해 2가지 핵심 변수가 필요하다.
* `ARCH=arm64`: 커널 소스 트리 내 `arch/arm64/` 디렉터리에 위치한 아키텍처 종속 코드 및 디바이스 트리(`.dts`)를 타겟으로 지정한다.
* `CROSS_COMPILE=aarch64-linux-gnu-`: `Kbuild`는 내부적으로 C 컴파일러를 호출할 때 `$(CROSS_COMPILE)gcc` 형태로 변수를 결합한다. 즉, 호스트의 기본 컴파일러를 배제하고 타겟 전용 컴파일러를 실행하도록 링킹 구조를 비튼다.

## 2. 파일 디스크립터(FD)와 `tee` 명령어의 앙상블
커널 빌드 시 발생하는 수만 줄의 로그를 효과적으로 관리하기 위해 `2>&1 | tee` 조합을 사용한다.

* 리눅스는 표준 입력(`0`), 정상 출력(`1`), 에러 출력(`2`)이라는 독립된 파일 디스크립터 통로를 가진다.
* `2>&1`은 에러 로그(`2`)의 출력 방향을 정상 로그(`1`) 통로로 병합하는 리다이렉션 기법이다.
* 파이프(`|`)로 전달받은 이 병합된 데이터를 `tee` 명령어가 수신하여, 마치 알파벳 'T' 모양 배관처럼 한 갈래는 터미널 화면에 뿌리고 다른 한 갈래는 지정된 로그 파일에 동시에 기록한다. 이를 통해 에러 발생 시 로그가 잘리는 현상을 방지한다.

## 3. 심볼릭 링크 폭탄의 회피: `scp -r` 대신 `tar`를 쓰는 이유
빌드된 커널 모듈(`/lib/modules/`)을 원격 보드로 전송할 때, 디렉터리를 그대로 복사하는 `scp -r`을 사용하면 치명적인 문제가 발생할 수 있다.
* 모듈 폴더 내부에는 호스트 PC의 커널 소스 트리 절대경로를 가리키는 `build`와 `source`라는 심볼릭 링크가 존재한다.
* `scp`는 이 링크를 단순 껍데기로 취급하지 않고 링크를 타고 들어가 원본 데이터(수 GB에 달하는 커널 전체 소스)를 통째로 타겟 보드로 복사하려 시도한다.
* 반면, `tar` 아카이브 유틸리티는 심볼릭 링크를 만나면 내용물을 따라가지 않고 링크 그 자체의 메타데이터만 얌전하게 압축하므로, 모듈 배포 시 반드시 `tar`를 이용한 압축 전송 기법을 적용해야 안전하다.
